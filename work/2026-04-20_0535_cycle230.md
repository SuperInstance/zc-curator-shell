## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive the following taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure (presets, LoRA export, tile handling)
- **holodeck** – Interactive simulation environments (MUD, NPCs, sentiment)
- **fleet-simulator** – Multi-agent simulation for training data generation
- **constraint-theory** – Geometric/relational constraint solving (snapping, rules)
- **cudaclaw** – GPU-resident agent runtime (CUDA, kernel fusion)
- **flux-runtime** – Self-assembling bytecode VM (WASM, hot-swapping)
- **shell** – Bootstrapping/capture algorithms (crab trap, classification)
- **ensign** – Specialist model exports (LoRA adapters, fine‑tuning)
- **vessel** – Agent‑specific workspaces (oracle1‑vessel, jetsonclaw1‑vessel)
- **docs** – Documentation, memory, protocol specs
- **tools** – Utilities, scripts, service management
- **research** – Exploratory code, experiments, rabbit‑trails
- **legacy** – Old prototypes, deprecated systems
- **staging** – In‑progress, not yet integrated
- **abandoned** – No recent activity, likely dead

**Tags (additional context):**
- `training` – Produces or consumes training data
- `simulation` – Simulates agents/environments
- `runtime` – Execution environment for agents
- `geometry` – Spatial/constraint solving
- `gpu` – CUDA/GPU‑accelerated
- `rust` – Written in Rust
- `python` – Written in Python
- `typescript` – Written in TypeScript
- `service` – Long‑running service (systemd, Docker)
- `cli` – Command‑line tool
- `api` – HTTP/WebSocket API
- `plugin` – Extensible plugin architecture
- `archive` – Historical record only
