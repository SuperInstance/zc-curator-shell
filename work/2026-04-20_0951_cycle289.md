## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive the following taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure (presets, LoRA export, tile management)
- **holodeck** – Interactive simulation environments (MUD server, NPCs)
- **fleet-simulator** – Multi-agent simulation for training data generation
- **constraint-theory** – Geometric/relational constraint solving (snapping, alignment)
- **cudaclaw** – GPU-resident agent runtime (kernel-level execution)
- **flux-runtime** – Self-assembling bytecode VM (dynamic code generation)
- **shell** – Bootstrapping/capture algorithms (crab trap, classification)
- **ensign** – Specialist model exports (LoRA adapters, fine‑tuned weights)
- **vessel** – Agent communication & state management (bottles, herald)
- **deadband** – Protocol implementation (P0/P1/P2 decision layers)
- **tools** – Utilities, APIs, benchmarks (model wrappers, eval scripts)
- **research** – Exploratory code, experiments, rabbit trails
- **docs** – Documentation, memory, brand assets
- **legacy** – Archived, deprecated, or inactive projects

**Tags (for cross‑cutting concerns):**
- `rust` – Rust implementation
- `python` – Python implementation
- `torch` – PyTorch based
- `training` – Involves model training
- `simulation` – Simulated environments
- `agent` – Multi‑agent systems
- `gpu` – GPU‑accelerated
- `edge` – Edge deployment (Jetson)
- `cloud` – Cloud services
- `cli` – Command‑line tool
- `api` – Web/network API
- `experimental` – High risk, unstable
- `stable` – Production‑ready
