## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive the following taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure (presets, LoRA export, tile handling)
- **holodeck** – Simulation environments (MUD server, NPCs, sentiment)
- **fleet-simulator** – Multi-agent simulation for training data
- **constraint-theory** – Geometric/relational constraint solving
- **cudaclaw** – GPU-resident agent runtime
- **flux-runtime** – Self-assembling bytecode VM
- **shell** – Bootstrapping/capture algorithms
- **ensign** – LoRA adapter export/management
- **deadband** – Protocol implementation (P0/P1/P2)
- **zeroclaw** – Agent framework (boot camp, tile submission)
- **vessel** – Agent state/communication (bottles, logs)
- **docs** – Documentation, memory, vision
- **tools** – Utilities, scripts, service management
- **research** – Experiments, prototypes, rabbit trails
- **legacy** – Old prototypes, deprecated code
- **external** – Forks, upstream dependencies, third-party integrations

**Tags (for finer granularity):**
- `training` – Produces or consumes training data
- `simulation` – Simulated environment
- `runtime` – Execution environment
- `protocol` – Communication/behavior standard
- `infrastructure` – Core platform service
- `agent` – Agent-specific logic
- `cli` – Command-line tool
- `api` – Web/network API
- `rust` / `python` / `cpp` – Primary language
- `wip` – Work in progress
- `archive` – Inactive but kept for reference
