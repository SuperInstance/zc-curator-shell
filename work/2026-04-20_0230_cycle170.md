## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive this working taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure (presets, LoRA export, tile handling)
- **holodeck** – Simulation environments (MUD server, NPCs, sentiment-aware worlds)
- **fleet-simulator** – Multi-agent simulation for training data generation
- **constraint-theory** – Geometric/relational constraint solving (snapping, alignment)
- **cudaclaw** – GPU-resident agent runtime (CUDA kernels, low-latency execution)
- **flux-runtime** – Self-assembling bytecode VM (modular execution environments)
- **shell** – Bootstrapping/capture algorithms (classify→score→complicate→capture)
- **ensign** – Specialist model exports (LoRA adapters, fine-tuned models)
- **zeroclaw** – Agent framework (task management, memory, tool use)
- **docs** – Documentation, specifications, protocol definitions
- **tools** – Utilities, scripts, CLI tools, health monitoring
- **research** – Exploratory code, experiments, rabbit trails
- **legacy** – Archived, deprecated, or inactive projects
- **staging** – In-progress, not yet integrated

**Tags (additional context):**
- `training` – Involves model training or data generation
- `simulation` – Simulated environments or agents
- `runtime` – Execution environment or VM
- `edge` – Edge computing/Jetson target
- `cloud` – Cloud deployment/lighthouse
- `rust` – Primary language Rust
- `python` – Primary language Python
- `torch` – Uses PyTorch
- `cuda` – Uses CUDA/GPU acceleration
- `proto` – Prototype/experimental
- `stable` – Production-ready
- `abandoned` – No recent activity, likely dead
