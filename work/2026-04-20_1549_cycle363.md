## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive the following taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure (presets, LoRA export, tile handling)
- **holodeck** – Interactive simulation environments (MUD, NPCs, sentiment)
- **fleet-simulator** – Multi-agent simulation for training data generation
- **constraint-theory** – Geometric/relational constraint solving (snapping, alignment)
- **cudaclaw** – GPU-resident agent runtime (kernel-level execution)
- **flux-runtime** – Self-assembling bytecode VM (dynamic code generation)
- **shell** – Bootstrapping/capture algorithms (crab trap, classification, scoring)
- **ensign** – Specialist model exports (LoRA adapters, fine‑tuned weights)
- **docs** – Documentation, memory, protocol specifications
- **tools** – Utilities, scripts, service management (service‑guard.sh, monitoring)
- **research** – Exploratory code, experiments, rabbit‑trail investigations
- **legacy** – Archived, deprecated, or inactive projects
- **staging** – In‑progress, unproven, or prototype work

**Tags (for cross‑cutting concerns):**
- `training` – Generates or consumes training data
- `simulation` – Simulates environments or agents
- `runtime` – Execution environment for agents
- `geometry` – Spatial/geometric operations
- `gpu` – GPU‑accelerated or CUDA‑based
- `rust` – Written in Rust
- `python` – Written in Python
- `torch` – Uses PyTorch
- `lora` – Involves LoRA adapters
- `protocol` – Defines communication or interaction protocols
- `monitoring` – Health checks, logging, observability
- `archive` – Read‑only, historical value only
