## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading the active fleet repos (plato-torch, plato-ensign, holodeck-rust, fleet-simulator, constraint-theory-core, cudaclaw, flux-runtime), I derive the following taxonomy:

**Primary Categories:**
- **plato** – PLATO room system, training, LoRA export
- **holodeck** – MUD server, NPC simulation, sentiment-aware environments
- **fleet** – multi-agent simulation, training data generation
- **constraint** – geometric snapping, constraint solving, CAD-like logic
- **cuda** – GPU-resident runtime, kernel-level agent execution
- **flux** – bytecode VM, self-assembling code, runtime composition
- **shell** – bootstrapping, agent capture, classification/scoring pipelines
- **tool** – utilities, CLI tools, devops, monitoring
- **research** – experiments, prototypes, rabbit‑trail explorations
- **archive** – deprecated, inactive, or historical reference

**Tags (for finer granularity):**
- `training` – produces or consumes training data
- `simulation` – synthetic environment or agent behavior
- `runtime` – execution environment for agents/models
- `geometry` – spatial reasoning, CAD, snapping
- `gpu` – CUDA, kernel, GPU‑resident
- `lora` – adapter training, fine‑tuning
- `mud` – multi‑user dungeon, text‑based world
- `protocol` – communication standard, API definition
- `monitor` – health checks, logging, observability
- `legacy` – old version, superseded, not actively maintained
