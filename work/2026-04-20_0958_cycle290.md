## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive this working taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure (presets, LoRA export, tile management)
- **holodeck** – Simulation environments (MUD server, NPCs, sentiment-aware worlds)
- **fleet-simulator** – Multi-agent simulation for training data generation
- **constraint-theory** – Geometric/relational constraint solving (snapping, alignment)
- **cudaclaw** – GPU-resident agent runtime (kernel fusion, persistent compute)
- **flux-runtime** – Self-assembling bytecode VM (modular execution)
- **shell** – Bootstrapping/capture algorithms (classify→score→complicate→capture)
- **ensign** – LoRA adapter management (export, merge, deploy)
- **vessel** – Agent communication & state persistence (bottles, message passing)
- **deadband** – Protocol implementation (P0/P1/P2 decision layers)
- **tools** – Developer utilities, CLI tools, API wrappers
- **research** – Exploratory code, experiments, rabbit trails
- **docs** – Documentation, memory, brand assets
- **legacy** – Archived, deprecated, or inactive projects

**Tags (additional context):**
- `training` – Produces or consumes training data
- `simulation` – Simulates agents or environments
- `runtime` – Execution environment for agents
- `protocol` – Defines communication or decision standards
- `infrastructure` – Foundational systems (rooms, shells, vessels)
- `experimental` – High-risk, unproven, or prototype stage
- `tool` – Utility for development or operations
- `archive` – Read‑only, historical reference
