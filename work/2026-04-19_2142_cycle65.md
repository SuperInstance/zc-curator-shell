## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive the following taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure, LoRA export, self‑training systems
- **holodeck** – MUD server, sentiment‑aware NPCs, interactive simulation
- **fleet‑simulator** – Multi‑agent simulation for training data generation
- **constraint‑theory** – Geometric snapping, constraint solving, CAD‑like logic
- **cudaclaw** – GPU‑resident agent runtime, low‑latency execution
- **flux‑runtime** – Self‑assembling bytecode VM, dynamic code generation
- **shell‑bootstrap** – Hermit‑crab algorithms for capturing external intelligence
- **mirror‑play** – I2I (image‑to‑image) training loops between PLATO sessions
- **deadband‑protocol** – P0/P1/P2 decision framework, negative‑space mapping
- **ensign** – Lightweight LoRA adapters exported from room experience
- **vessel** – Agent‑specific workspaces, state persistence, message bottles
- **tooling** – CLI utilities, health checks, service guards, dev‑ops scripts
- **research** – Rabbit‑trail explorations, concept prototypes, academic notes
- **brand** – Logos, wordmarks, visual identity, architectural‑as‑brand assets
- **documentation** – READMEs, protocols, memory files, training tiles

**Status Tags:**
- **active** – In active development, part of the live fleet
- **stale** – No recent commits, but potentially reusable
- **abandoned** – Deprecated, superseded, or orphaned
- **experiment** – One‑off test, proof‑of‑concept, not production‑ready
- **archive** – Historical record, kept for reference only
