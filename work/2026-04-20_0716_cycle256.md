## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive the following taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure (presets, LoRA export, tile handling)
- **holodeck** – Interactive simulation environments (MUD server, NPCs)
- **fleet-simulator** – Multi-agent simulation for training data generation
- **constraint-theory** – Geometric/relational constraint systems (snapping, solving)
- **cudaclaw** – GPU-resident agent runtime (kernel fusion, low-latency execution)
- **flux-runtime** – Self-assembling bytecode VM (dynamic reconfiguration)
- **shell** – Bootstrapping/capture algorithms (crab trap, classification, scoring)
- **ensign** – Specialist model training (LoRA adapters, room experience export)
- **vessel** – Agent communication & state management (bottles, message passing)
- **tools** – Developer utilities, APIs, benchmarks, CLI tools
- **research** – Exploratory code, experiments, rabbit trails
- **docs** – Documentation, memory, protocol specifications
- **legacy** – Archived, deprecated, or inactive projects
- **stub** – Empty, placeholder, or very early-stage repos

**Tags (for finer granularity):**
- `training` – Involves model training or data generation
- `simulation` – Simulated environments or agents
- `runtime` – Execution environment or VM
- `geometry` – Spatial/geometric computations
- `gpu` – GPU‑accelerated or CUDA code
- `rust` – Primary language Rust
- `python` – Primary language Python
- `typescript` – Primary language TypeScript
- `agent` – Multi‑agent system component
- `protocol` – Communication or interaction protocol
- `archive` – Historical record only
- `wip` – Work in progress, incomplete
