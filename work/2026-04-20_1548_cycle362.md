## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive the following taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure, LoRA export, self‑training systems
- **holodeck** – MUD server, sentiment‑aware NPCs, interactive simulation
- **fleet‑simulator** – Multi‑agent simulation for generating training data
- **constraint‑theory** – Geometric snapping, spatial reasoning, rule‑based constraints
- **cudaclaw** – GPU‑resident agent runtime, low‑latency execution
- **flux‑runtime** – Self‑assembling bytecode VM, dynamic code generation
- **shell‑bootstrap** – Bootstrapping algorithms, external intelligence capture
- **deadband‑protocol** – P0/P1/P2 decision framework, negative‑space mapping
- **agent‑runtime** – Agent execution environment, message‑passing, tool‑calling
- **research‑trail** – Exploratory code, experiments, one‑off prototypes
- **docs‑brand** – Documentation, branding assets, identity materials
- **tool‑integration** – External tool wrappers, API clients, utility scripts
- **legacy‑archive** – Old prototypes, deprecated systems, historical artifacts

**Tags (for cross‑cutting concerns):**
- `rust` – Implemented in Rust
- `python` – Implemented in Python
- `torch` – Uses PyTorch
- `training‑data` – Generates or processes training data
- `simulation` – Contains simulation logic
- `gpu` – GPU‑accelerated
- `edge` – Edge‑deployable (Jetson, etc.)
- `cloud` – Cloud‑native / serverless
- `cli` – Command‑line tool
- `web` – Web interface / API
- `experimental` – High risk, unstable, proof‑of‑concept
- `stable` – Production‑ready, tested
- `abandoned` – No recent activity, likely dead
