## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive the following taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure, LoRA export, self‑training algorithms
- **holodeck** – Multi‑user dungeon (MUD) server, sentiment‑aware NPCs, interactive simulation
- **fleet‑simulator** – Fleet‑level simulation for generating training data
- **constraint‑theory** – Geometric snapping, constraint solving, spatial reasoning
- **cudaclaw** – GPU‑resident agent runtime, low‑latency execution
- **flux‑runtime** – Self‑assembling bytecode VM, dynamic code generation
- **shell‑bootstrap** – Bootstrapping algorithms for capturing external intelligence
- **deadband‑protocol** – P0/P1/P2 decision framework, safety‑first navigation
- **agent‑orchestration** – Multi‑agent coordination, communication protocols
- **tools‑integrations** – API wrappers, model adapters, external service connectors
- **docs‑knowledge** – Documentation, memory, training materials, brand assets
- **experimental** – Proof‑of‑concept, research, untested ideas
- **abandoned‑stale** – Inactive, unmaintained, deprecated

**Tags (for finer granularity):**
- `training` – produces or consumes training data
- `simulation` – synthetic environment
- `runtime` – execution environment
- `geometry` – spatial/math operations
- `protocol` – communication/decision standard
- `rust` – Rust implementation
- `python` – Python implementation
- `cuda` – GPU‑accelerated
- `edge` – Jetson/embedded focus
- `cloud` – cloud‑native/lighthouse
- `research` – exploratory, not production
- `tool` – utility, helper script
