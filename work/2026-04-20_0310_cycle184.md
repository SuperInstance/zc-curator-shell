## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive the following taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure (presets, LoRA export, tile servers)
- **holodeck** – Interactive simulation environments (MUD, NPCs, sentiment)
- **fleet-simulator** – Multi-agent simulation for training data generation
- **constraint-theory** – Geometric/relational constraint solving (snapping, alignment)
- **cudaclaw** – GPU-resident agent runtime (CUDA kernels, low-latency execution)
- **flux-runtime** – Self-assembling bytecode VM (dynamic code generation)
- **shell** – Bootstrapping/capture algorithms (classify→score→complicate→capture)
- **ensign** – Specialist model exports (LoRA adapters, fine‑tuned weights)
- **deadband** – Protocol implementation (P0/P1/P2 decision layers)
- **tools** – Developer utilities, CLI tools, API wrappers
- **research** – Experimental code, rabbit‑trail explorations, concept proofs
- **docs** – Documentation, specifications, architecture notes
- **brand** – Visual identity, logos, wordmarks, brand assets
- **legacy** – Old prototypes, deprecated code, historical experiments

**Tags (for cross‑cutting concerns):**
- `rust` – Written in Rust
- `python` – Written in Python
- `torch` – Uses PyTorch
- `cuda` – Uses CUDA/GPU kernels
- `simulation` – Simulates agents/environments
- `training` – Produces or consumes training data
- `api` – Provides an API/service
- `cli` – Command‑line tool
- `experimental` – High‑risk, unstable, research‑grade
- `stable` – Production‑ready, tested
- `deprecated` – No longer maintained
