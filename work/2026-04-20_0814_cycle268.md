## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive the following taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure (presets, LoRA export, tile servers)
- **holodeck** – Interactive simulation environments (MUD, NPCs, sentiment)
- **fleet-simulator** – Multi-agent simulation for training data generation
- **constraint-theory** – Geometric/relational constraint solving (snapping, CAD)
- **cudaclaw** – GPU-resident agent runtime (CUDA kernels, low-latency execution)
- **flux-runtime** – Self-assembling bytecode VM (dynamic code generation)
- **shell** – Bootstrapping/capture algorithms (classify→score→complicate→capture)
- **ensign** – Specialist model training (LoRA adapters, room experience export)
- **oracle** – Cloud lighthouse services (long‑running, coordination, memory)
- **edge** – Jetson/embedded deployment (bare‑metal, real‑time operation)
- **tooling** – Development utilities, scripts, CLI tools
- **research** – Exploratory code, experiments, rabbit‑trail documents
- **documentation** – READMEs, protocols, vision statements, memory logs
- **brand** – Logos, wordmarks, visual identity assets
- **legacy** – Old prototypes, deprecated code, historical artifacts

**Tags (for cross‑cutting concerns):**
- `rust` | `python` | `cuda` | `torch` | `docker` | `wasm`
- `training` | `simulation` | `runtime` | `agent` | `geometry` | `visualization`
- `active` | `stale` | `abandoned` | `experimental` | `production‑ready`
