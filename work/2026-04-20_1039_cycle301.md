## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive the following taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure, LoRA export, self‑training algorithms
- **holodeck** – MUD server, sentiment‑aware NPCs, interactive simulation
- **fleet‑simulator** – Multi‑agent simulation for training data generation
- **constraint‑theory** – Geometric snapping, spatial reasoning, constraint solving
- **cudaclaw** – GPU‑resident agent runtime, kernel‑level execution
- **flux‑runtime** – Self‑assembling bytecode VM, dynamic code generation
- **shell‑bootstrapping** – Algorithms to capture external agent intelligence
- **deadband‑protocol** – P0/P1/P2 decision framework, negative‑space mapping
- **agent‑orchestration** – Multi‑agent coordination, communication protocols
- **tool‑integration** – API wrappers, model connectors, external service adapters
- **research‑trails** – Exploratory code, experiments, prototypes
- **docs‑assets** – Documentation, branding, memory, training materials

**Tags (for finer granularity):**
- `training` – Produces or consumes training data
- `simulation` – Simulates environments or agents
- `runtime` – Execution environment for agents
- `geometry` – Spatial/mathematical reasoning
- `protocol` – Communication or decision standards
- `rust` – Primary language Rust
- `python` – Primary language Python
- `torch` – Uses PyTorch
- `lora` – Involves LoRA adapters
- `gpu` – GPU‑accelerated
- `edge` – Edge‑deployment focused
- `cloud` – Cloud‑service oriented
- `legacy` – Older, possibly deprecated
- `stub` – Minimal placeholder
- `abandoned` – No recent activity, likely dead
