## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive the following taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure (presets, LoRA export, tile handling)
- **holodeck** – Interactive simulation environments (MUD server, NPCs)
- **fleet-simulator** – Multi-agent simulation for training data generation
- **constraint-theory** – Geometric/relational constraint solving (snapping, rules)
- **cudaclaw** – GPU-resident agent runtime (CUDA kernels, low‑latency execution)
- **flux-runtime** – Self‑assembling bytecode VM (dynamic code generation)
- **shell** – Bootstrapping/capture algorithms (classify→score→complicate→capture)
- **ensign** – LoRA adapter export & management (specialist model creation)
- **mirror-play** – Iterative screen‑to‑screen training (I2I data generation)
- **deadband** – Protocol implementation (P0/P1/P2 decision layers)
- **tools** – Developer utilities, CLI tools, API wrappers
- **docs** – Documentation, memory, vision, protocol specs
- **research** – Exploratory code, experiments, rabbit‑trails
- **abandoned** – Inactive, deprecated, or archived projects
- **stub** – Minimal placeholder, not yet developed

**Tags (for cross‑cutting concerns):**
- `rust` – Implemented in Rust
- `python` – Implemented in Python
- `torch` – Uses PyTorch
- `cuda` – GPU‑accelerated
- `training` – Produces or consumes training data
- `simulation` – Simulates agents/environments
- `protocol` – Implements a communication or decision protocol
- `infra` – Infrastructure (servers, orchestration, monitoring)
- `cli` – Command‑line tool
- `web` – Web interface or API
