## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive this working taxonomy:

**Core Domains:**
- `plato` — Training rooms, LoRA exports, self‑training infrastructure
- `holodeck` — MUD servers, NPCs, sentiment‑aware simulation
- `fleet-simulator` — Multi‑agent simulation for training data
- `constraint-theory` — Geometric snapping, constraint solving
- `cudaclaw` — GPU‑resident agent runtime
- `flux-runtime` — Self‑assembling bytecode VM
- `shell` — Bootstrapping, capture algorithms, crab‑trap logic
- `mirror-play` — I2I iteration, screen‑to‑screen training

**Support Categories:**
- `tools` — CLI utilities, scripts, dev tools
- `docs` — Documentation, specs, RFCs
- `research` — Experiments, prototypes, rabbit‑trails
- `archive` — Inactive, deprecated, or merged work
- `asset` — Brand assets, logos, imagery
- `meta` — Fleet management, agent coordination, task boards

**Status Tags:**
- `active` — Actively developed, part of current fleet
- `stale` — No recent commits, may be dormant
- `abandoned` — Unmaintained, likely dead‑end
- `merged` — Functionality absorbed into another repo
- `seed` — Early prototype, not yet integrated
