## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive the following taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure (presets, LoRA export, tile management)
- **holodeck** – Interactive simulation environments (MUD server, NPCs)
- **fleet-simulator** – Multi-agent simulation for training data generation
- **constraint-theory** – Geometric/relational constraint solving (snapping, alignment)
- **cudaclaw** – GPU-resident agent runtime (CUDA kernels, low-latency execution)
- **flux-runtime** – Self-assembling bytecode VM (dynamic code generation)
- **shell** – Bootstrapping/capture algorithms (classify→score→complicate→capture)
- **ensign** – Specialist model exports (LoRA adapters, fine‑tuned weights)
- **tools** – Development utilities, CLI tools, API wrappers
- **research** – Experimental code, rabbit‑trail explorations, prototypes
- **docs** – Documentation, specifications, protocol definitions
- **ops** – Deployment, monitoring, service management (Docker, k8s, scripts)
- **brand** – Visual identity, logos, wordmarks, brand assets
- **archive** – Inactive, deprecated, or historical projects

**Tags (for finer granularity):**
- `training` – Involves model training or data generation
- `simulation` – Simulated environments or agents
- `runtime` – Execution environment for agents/models
- `geometry` – Spatial reasoning, snapping, constraints
- `gpu` – CUDA/GPU‑accelerated code
- `rust` – Primary language is Rust
- `python` – Primary language is Python
- `torch` – Uses PyTorch
- `api` – Provides an API or service interface
- `cli` – Command‑line tool
- `web` – Web interface or frontend
- `experimental` – High‑risk, unstable, proof‑of‑concept
- `stable` – Production‑ready, tested
- `deprecated` – No longer maintained
