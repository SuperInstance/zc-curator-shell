## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive this working taxonomy:

**Core Domains:**
- **plato** – Training room infrastructure (presets, LoRA export, tile handling)
- **holodeck** – Simulation environments (MUD server, NPCs, sentiment)
- **fleet-simulator** – Multi-agent simulation for training data generation
- **constraint-theory** – Geometric/relational constraint solving (snapping, alignment)
- **cudaclaw** – GPU-resident agent runtime (CUDA kernels, low-latency execution)
- **flux-runtime** – Self-assembling bytecode VM (modular, hot-swappable)
- **shell** – Bootstrapping/capture algorithms (classify→score→complicate→capture)
- **ensign** – Specialist model exports (LoRA adapters, fine‑tuned weights)
- **vessel** – Agent‑specific workspaces (oracle1‑vessel, jetsonclaw1‑vessel)
- **research** – Exploratory code, experiments, rabbit‑trails
- **tooling** – Development utilities, scripts, service guards
- **docs** – Documentation, memory, protocol definitions
- **brand** – Logos, wordmarks, visual identity assets

**Tags (for finer granularity):**
- `training` – Involves model training or data generation
- `simulation` – Simulated environments or agents
- `runtime` – Execution environment / VM
- `geometry` – Spatial/geometric operations
- `gpu` – CUDA/GPU‑accelerated code
- `rust` – Primary language Rust
- `python` – Primary language Python
- `torch` – Uses PyTorch
- `lora` – LoRA adapter creation/management
- `agent` – Multi‑agent system component
- `service` – Long‑running service (daemon)
- `cli` – Command‑line tool
- `archive` – Inactive, historical, or deprecated
