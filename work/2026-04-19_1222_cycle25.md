From reading fleet-knowledge/ and active fleet repos, I derive the following taxonomy:

- **plato** – Training room infrastructure (presets, LoRA export, tile management)
- **holodeck** – Simulation environments (MUD server, NPCs, sentiment-aware worlds)
- **fleet-simulator** – Multi-agent simulation for training data generation
- **constraint-theory** – Geometric/relational constraint solving (snapping, alignment)
- **cudaclaw** – GPU-resident agent runtime (CUDA kernels, low-latency execution)
- **flux-runtime** – Self-assembling bytecode VM (modular, hot-swappable)
- **shell** – Bootstrapping algorithms (classify→score→complicate→capture)
- **mirror-play** – Iterative image-to-image LoRA training (two PLATO sessions)
- **deadband-protocol** – P0/P1/P2 decision framework (negative space mapping)

**Support Categories:**
- **docs** – Documentation, specifications, protocol definitions
- **tools** – CLI utilities, service scripts, health monitors
- **research** – Exploratory code, rabbit trails, concept proofs
- **archive** – Inactive, deprecated, or historical projects
- **assets** – Branding, logos, icons, visual identity
- **training** – Boot camp materials, tutorials, agent onboarding

**Tags (for cross-cutting concerns):**
- `rust` – Implemented in Rust
- `python` – Implemented in Python
- `torch` – Uses PyTorch
- `cuda` – GPU-accelerated
- `agent` – Multi-agent system component
- `simulation` – Simulated environment
- `training` – Generates or consumes training data
- `infrastructure` – Core platform service
