## 1. Fleet Taxonomy (Derived from Active Fleet Repos)

From reading fleet-knowledge/ and active fleet repos, I derive this working taxonomy:

**Core Domains:**
- `plato` – Training rooms, LoRA export, self‑training infrastructure
- `holodeck` – MUD server, sentiment‑aware NPCs, interactive simulation
- `fleet-simulator` – Multi‑agent simulation for training data generation
- `constraint-theory` – Geometric snapping, constraint solving (Rust)
- `cudaclaw` – GPU‑resident agent runtime, kernel‑level execution
- `flux-runtime` – Self‑assembling bytecode VM, dynamic code generation
- `shell` – Bootstrapping, capture‑classify‑score‑complicate algorithms
- `mirror-play` – I2I (image‑to‑image) training, dual‑PLATO LoRA data
- `deadband` – Protocol implementation (P0/P1/P2), safety layers
- `zeroclaw` – Agent runtime, tile submission, validation
- `tools` – Utilities, scripts, service management (e.g., service‑guard.sh)
- `research` – Exploratory code, rabbit‑trail documents, prototypes
- `docs` – Documentation, memory, task boards, architecture notes
- `brand` – Logos, wordmarks, visual assets, identity guidelines
- `legacy` – Old experiments, deprecated code, historical artifacts

**Tags (additional context):**
- `rust` / `python` / `cpp` – Primary language
- `training` – Produces or consumes training data
- `simulation` – Simulates agents or environments
- `runtime` – Execution environment for agents
- `protocol` – Communication or behavioral specification
- `infrastructure` – Foundational service (e.g., PLATO server)
- `agent` – Implements an agent or agent component
- `edge` – Jetson/Orin target, bare‑metal focus
- `cloud` – Cloud‑native, lighthouse services
- `gpu` – CUDA/GPU‑accelerated
- `wip` – Work in progress, incomplete
- `abandoned` – No recent activity, likely dead
